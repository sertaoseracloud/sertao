## Summary

Phase 6 adds three engagement capabilities to an Astro 6 SSG blog: Giscus comment embeds (GitHub Discussions-backed), Pagefind full-text search with a `Ctrl+K` modal, and tag browse pages. All three are zero-backend: Giscus runs as a CDN iframe; Pagefind generates a static index at build time; tag pages are generated by `getStaticPaths` at build.

The primary technical complexity is understanding the correct Astro patterns for each integration. Two important corrections from the CONTEXT.md assumptions are documented below: `client:visible` does NOT apply to plain Astro components (confirmed by Astro docs and project history), and CommentsEmbed must use `is:inline` on its Giscus script tag rather than the outer component carrying `client:visible`. Additionally, the `@pagefind/default-ui` npm package is the correct initialization path, but the runtime loading comes from `/pagefind/pagefind-ui.js` at the path generated by the CLI â€” requiring a specific dynamic import pattern when used from a bundled Astro `<script>` block.

**Primary recommendation:** Follow the UI-SPEC.md exactly â€” it already embodies the correct patterns. The one adjustment needed is around `CommentsEmbed`: the `client:visible` directive in the CONTEXT.md refers to a framework component pattern, but since CommentsEmbed is a plain Astro component, the correct lazy approach is either (a) remove `client:visible` and rely on `data-loading="lazy"` on the Giscus script tag, or (b) implement an IntersectionObserver in the component's plain `<script>` block. The `data-loading="lazy"` attribute on the Giscus `<script>` tag is the simpler and officially supported path.

## Pattern 1: `client:visible` Does NOT Apply to Plain Astro Components

**What:** The `client:visible` directive hydrates a UI framework component (React/Vue/Svelte) when it scrolls into view. Plain Astro `.astro` components with a `<script>` block are NOT framework components â€” their scripts run as bundled client JS automatically. Applying `client:visible` to a plain Astro component is invalid syntax and produces a build warning. [VERIFIED: Astro docs + project STATE.md confirms "client:visible removed from CopyCode" in Phase 4]

**Consequence for CommentsEmbed (D-03/D-05):** The CONTEXT.md says `<CommentsEmbed client:visible>`. This is the wrong approach for a plain Astro component. The correct lazy-loading pattern for CommentsEmbed is to use **`data-loading="lazy"` on the Giscus `<script>` tag itself**. Giscus natively supports this attribute and defers iframe creation until the section scrolls into view.

**Pattern:**
```astro
<!-- CommentsEmbed.astro â€” plain Astro component, NO client: directive -->
<script
  is:inline
  src="https://giscus.app/client.js"
  data-repo="sertaoseracloud/sertao"
  data-repo-id="{REPO_ID}"
  data-category="Comments"
  data-category-id="{CATEGORY_ID}"
  data-mapping="og:title"
  data-strict="0"
  data-reactions-enabled="1"
  data-emit-metadata="0"
  data-input-position="top"
  data-theme="dark"
  data-lang="pt"
  data-loading="lazy"
  crossorigin="anonymous"
  async
></script>
```

**PostLayout usage:**
```astro
<!-- No client: directive â€” CommentsEmbed is a plain Astro component -->
<CommentsEmbed />
```

**Note:** Giscus `<script>` tags MUST use `is:inline` in Astro. Without `is:inline`, Astro would bundle the script and strip its `data-*` attributes, breaking Giscus configuration. [VERIFIED: multiple community integrations confirmed this, e.g., ailinklab.com 2025]

## Pattern 2: `@pagefind/default-ui` â€” Import from npm in Bundled Script

**What:** Two approaches exist. The CONTEXT.md uses `import('/pagefind/pagefind-ui.js')` (dynamic runtime load). An equivalent approach uses `import { PagefindUI } from "@pagefind/default-ui"` inside an Astro `<script>` block â€” Astro bundles this, and since `@pagefind/default-ui` re-exports the same `PagefindUI` class, both work.

**The recommended pattern for this project** (per CONTEXT.md D-07, UI-SPEC.md pattern) uses dynamic import of the CDN-generated file. This avoids bundling the UI library (which doesn't exist at dev time) into the Astro bundle:

```typescript
// Inside Search.astro <script> block
let pagefindInitialized = false;

function openModal() {
  const modal = document.getElementById('search-modal') as HTMLElement;
  modal.style.display = 'block';
  if (!pagefindInitialized) {
    pagefindInitialized = true;
    import('/pagefind/pagefind-ui.js').then(({ PagefindUI }) => {
      new PagefindUI({
        element: '#pagefind-search',
        showSubResults: false,
        translations: {
          placeholder: 'Buscar posts...',
          zero_results: 'Nenhum resultado para "[SEARCH_TERM]"',
        },
      });
    });
  }
  // Move focus to search input after Pagefind renders
  setTimeout(() => {
    const input = document.querySelector<HTMLInputElement>('#pagefind-search input');
    input?.focus();
  }, 50);
}
```

**CSS for Pagefind UI:** Load from the CDN-generated file via a `<link>` tag in the Search component or BaseLayout:
```astro
<!-- Alternatively, load in Search.astro component head if needed -->
<link rel="stylesheet" href="/pagefind/pagefind-ui.css" />
```
OR import via npm in the component's `<script>` (requires `@pagefind/default-ui` installed):
```typescript

```

The UI-SPEC.md correctly uses the CSS variable override approach via the `#pagefind-search` element selector. [CITED: pagefind.app/docs/ui-usage/]

## Pattern 3: Giscus `postMessage` Theme Bridge

**What:** When ThemeToggle changes the page theme, it must notify the Giscus iframe to update its own theme. Giscus supports this via `postMessage` to `https://giscus.app`.

**Selector:** `iframe.giscus-frame` â€” the class Giscus assigns to its iframe element. [VERIFIED: giscus ADVANCED-USAGE.md]

**Message format:**
```typescript
function sendGiscusTheme(theme: 'dark' | 'light'): void {
  const giscusFrame = document.querySelector<HTMLIFrameElement>('iframe.giscus-frame');
  if (giscusFrame?.contentWindow) {
    giscusFrame.contentWindow.postMessage(
      { giscus: { setConfig: { theme } } },
      'https://giscus.app'
    );
  }
}
```

**Safety:** `?.contentWindow` optional chain means this is a no-op on pages without Giscus. Add this call at the END of the existing `applyTheme()` function in `ThemeToggle.astro`. [VERIFIED: giscus ADVANCED-USAGE.md]

**Giscus built-in themes:** `dark`, `light`, `dark_dimmed`, `dark_high_contrast`, `light_high_contrast`, etc. For this project, `'dark'` and `'light'` suffice. [CITED: giscus.app documentation]

## Pattern 4: Giscus Script â€” `is:inline` Requirement

**What:** In Astro, `<script>` tags inside components are bundled and processed by Vite unless `is:inline` is specified. The Giscus loader at `https://giscus.app/client.js` reads `data-*` attributes from the originating `<script>` tag at runtime â€” this requires the `<script>` tag to exist as-written in the final HTML DOM.

**Without `is:inline`:** Astro bundles the script, the `data-*` attributes are stripped from the bundle, and Giscus cannot find its configuration. The script becomes a dead empty bundle entry.

**With `is:inline`:** Astro outputs the `<script>` tag as literal HTML with all `data-*` attributes preserved. Giscus reads them correctly.

```astro
<!-- CORRECT -->
<script is:inline src="https://giscus.app/client.js" data-repo="..." ...></script>

<!-- WRONG â€” Astro bundles this and strips data-* attrs -->
<script src="https://giscus.app/client.js" data-repo="..." ...></script>
```

[CITED: ailinklab.com/post/2025/2025-01-11/integrate-giscus-to-astro-blog/ + common Astro pattern]

## Pattern 5: Tag Pages with `getStaticPaths`

**What:** `/tags/[tag].astro` uses the same `getCollection` + `getStaticPaths` pattern as `src/pages/posts/[...slug].astro`. With `legacy.collectionsBackwardsCompat: true` already in `astro.config.mjs`, the collection API works identically.

**getStaticPaths implementation:**
```typescript
export async function getStaticPaths() {
  const posts = await getCollection('posts', ({ data }) =>
    import.meta.env.PROD ? !data.draft : true,
  );

  // Collect all unique normalized tags
  const tagMap = new Map<string, typeof posts>();
  for (const post of posts) {
    for (const raw of (post.data.tags ?? [])) {
      const tag = raw.toLowerCase();
      if (!tagMap.has(tag)) tagMap.set(tag, []);
      tagMap.get(tag)!.push(post);
    }
  }

  return Array.from(tagMap.entries()).map(([tag, tagPosts]) => ({
    params: { tag },
    props: { posts: tagPosts.sort((a, b) => b.data.pubDate.valueOf() - a.data.pubDate.valueOf()) },
  }));
}
```

**Tag index page data aggregation:**
```typescript
const posts = await getCollection('posts', ({ data }) =>
  import.meta.env.PROD ? !data.draft : true,
);

const tagCounts = new Map<string, number>();
for (const post of posts) {
  const seen = new Set<string>();
  for (const raw of (post.data.tags ?? [])) {
    const tag = raw.toLowerCase();
    if (!seen.has(tag)) {
      seen.add(tag);
      tagCounts.set(tag, (tagCounts.get(tag) ?? 0) + 1);
    }
  }
}

const sortedTags = Array.from(tagCounts.entries())
  .sort(([a, ca], [b, cb]) => cb - ca || a.localeCompare(b));
```

## Pattern 7: Pagefind Build Integration

**What:** Pagefind indexes the built `dist/` folder after `astro build` completes. The CLI is `pagefind --site dist` and writes static index files to `dist/pagefind/`.

**`package.json` change:**

**`pagefind --site dist` behavior:**
- Crawls all `dist/**/*.html` files
- Reads text content from `<body>` (excluding `data-pagefind-ignore` elements)
- Extracts page title and description from `<head>`
- Writes `dist/pagefind/pagefind.js`, `dist/pagefind/pagefind-ui.js`, `dist/pagefind/pagefind-ui.css`, and index shards

## Common Pitfalls

### Pitfall 1: `client:visible` on Plain Astro Component
**What goes wrong:** Adding `client:visible` to `<CommentsEmbed />` in PostLayout. Astro throws a build warning: "client:* directives are not compatible with Astro components." The component renders but the directive is silently ignored â€” no lazy loading occurs.
**Why it happens:** `client:*` directives are for framework components (React/Svelte/Vue islands). Plain Astro components have no hydration boundary.
**How to avoid:** Use `data-loading="lazy"` on the Giscus `<script is:inline>` tag â€” this is Giscus's native lazy-loading mechanism and defers iframe creation until the element scrolls into view. Remove any `client:visible` from the component usage.
**Warning signs:** Build output logs contain "client:visible is not supported on Astro components".

### Pitfall 2: Giscus Script Without `is:inline`
**What goes wrong:** Writing `<script src="https://giscus.app/client.js" data-repo="...">` without `is:inline`. Astro strips all `data-*` attributes during bundling. Giscus loads but finds no configuration and renders a broken/empty widget.
**Why it happens:** Astro bundles `<script>` tags and treats external URLs as module imports to process.
**How to avoid:** Always use `is:inline` on the Giscus script tag.
**Warning signs:** Giscus iframe loads but shows a configuration error. Network tab shows giscus.app/client.js loading but the iframe element has no `data-*` attributes.

### Pitfall 3: Pagefind UI Not Available During Dev
**What goes wrong:** `import('/pagefind/pagefind-ui.js')` fails at `http://localhost:4321/pagefind/pagefind-ui.js` with 404 because `dist/pagefind/` doesn't exist during development.
**Why it happens:** Pagefind index is generated at build time, not served by `astro dev`.
**How to avoid:** Accept this limitation â€” search only works after a build. Alternatively (advanced): copy `dist/pagefind/` to `public/pagefind/` after a local build, but this is not required for the Phase 6 scope.
**Warning signs:** Search modal opens but results never load; browser console shows 404 for `/pagefind/pagefind-ui.js`.

### Pitfall 4: Giscus `data-repo` Contains the Wrong Repo
**What goes wrong:** Using the wrong value for `data-repo`. The CONTEXT says to use `sertaoseracloud/sertao` but the actual GitHub user is `sertaoseracloud` â€” the repo name may differ.
**Why it happens:** The repo must have GitHub Discussions enabled, and the `data-repo`, `data-repo-id`, and `data-category-id` must match exactly.
**How to avoid:** Obtain `data-repo-id` and `data-category-id` from [https://giscus.app](https://giscus.app) after enabling GitHub Discussions. These IDs are unique to the repo and cannot be guessed.
**Warning signs:** Giscus iframe shows "Discussion not found" or "Repository not found".

### Pitfall 5: Tag Normalization Applied at Render but Not in `getStaticPaths`
**What goes wrong:** Tag chips on cards navigate to `/tags/AWS` but `getStaticPaths` only generates `/tags/aws`. The link leads to a 404.
**Why it happens:** Normalization (`.toLowerCase()`) applied in one place but not the other.
**How to avoid:** Apply `.toLowerCase()` in three places consistently: (1) `getStaticPaths` param generation, (2) tag chip `href` construction in all card/post renderers, (3) tag count aggregation in `/tags/index.astro`.
**Warning signs:** Tag chip clicks result in 404 pages.

### Pitfall 6: `postMessage` Theme Bridge Fires Before Giscus Iframe Loads
**What goes wrong:** The `applyTheme()` call on mount corrects the icon for stored/system preference, but at that moment the Giscus iframe hasn't been created yet (it's lazy). The `postMessage` fires but `iframe.giscus-frame` returns `null`. Then when the user scrolls and Giscus loads with `data-theme="dark"`, if the current theme is `light`, the comment section appears in dark mode.
**Why it happens:** `CommentsEmbed` is lazy (`data-loading="lazy"`) â€” the iframe doesn't exist at page load. `ThemeToggle.applyTheme()` runs on mount to sync the icon, calling the postMessage which silently no-ops.
**How to avoid:** The optional chain `?.contentWindow` already handles the null case silently. The Giscus iframe initial `data-theme` attribute should match the default theme. Since the project defaults to dark mode (no `data-theme` = dark in the CSS), and `data-theme="dark"` is set in CommentsEmbed, the initial state is correct for dark. When a user has stored `light` preference, `applyTheme('light')` fires on mount â€” `postMessage` no-ops (iframe not yet created) â€” but when the user scrolls to comments, Giscus loads with `data-theme="dark"` from the static attribute. This is a minor mismatch only in the light-preference case.
**Fix:** In CommentsEmbed, set `data-theme` dynamically at mount time based on `localStorage.getItem('theme') || 'dark'` using an inline script, OR simply accept the minor visual flash (Giscus will correct on next user toggle).
**Recommended approach:** Add a small `<script is:inline>` in CommentsEmbed that reads `localStorage.theme` and sets the correct `data-theme` on the sibling script tag before Giscus loads:
```html
<script is:inline>
  (function() {
    const theme = localStorage.getItem('theme') || 'dark';
    const giscusScript = document.currentScript?.previousElementSibling;
    if (giscusScript) giscusScript.setAttribute('data-theme', theme);
  })();
</script>
```
This is optional for MVP â€” the flash is brief and only affects light-mode users.

### Pitfall 7: `pagefind --site dist` in GitHub Actions
**What goes wrong:** The GitHub Actions deploy workflow runs `pnpm build` which now includes `pagefind --site dist`. If `pagefind` binary is not installed in CI, the build fails.
**Why it happens:** `pagefind` is a native binary. When installed as a devDependency via npm, it downloads the platform-specific binary via a postinstall script.
**How to avoid:** `pagefind` is added as a devDependency to `package.json` (`pnpm add -D pagefind`). This means `pnpm install` in CI will install the binary. The existing `.github/workflows/deploy.yml` already runs `pnpm install` before `pnpm build` â€” no changes to the CI workflow are needed.
**Warning signs:** CI fails with "pagefind: command not found" after the `pnpm build` step.

### Search.astro â€” modal + Pagefind init (core pattern)
```typescript
// Inside <script> block of Search.astro
const toggleBtn = document.getElementById('search-toggle') as HTMLButtonElement;
const modal = document.getElementById('search-modal') as HTMLElement;
let pagefindInitialized = false;

function openSearch(): void {
  modal.style.display = 'block';
  toggleBtn.setAttribute('aria-label', 'Fechar busca');
  if (!pagefindInitialized) {
    pagefindInitialized = true;
    import('/pagefind/pagefind-ui.js').then(({ PagefindUI }) => {
      new PagefindUI({
        element: '#pagefind-search',
        showSubResults: false,
        translations: {
          placeholder: 'Buscar posts...',
          zero_results: 'Nenhum resultado para "[SEARCH_TERM]"',
        },
      });
      setTimeout(() => {
        document.querySelector<HTMLInputElement>('#pagefind-search input')?.focus();
      }, 50);
    });
  } else {
    document.querySelector<HTMLInputElement>('#pagefind-search input')?.focus();
  }
}

function closeSearch(): void {
  modal.style.display = 'none';
  toggleBtn.setAttribute('aria-label', 'Abrir busca');
  toggleBtn.focus();
}

toggleBtn.addEventListener('click', () => {
  modal.style.display === 'none' ? openSearch() : closeSearch();
});

document.addEventListener('keydown', (e) => {
  if ((e.ctrlKey || e.metaKey) && e.key === 'k') {
    e.preventDefault();
    modal.style.display === 'none' ? openSearch() : closeSearch();
  }
  if (e.key === 'Escape' && modal.style.display !== 'none') {
    closeSearch();
  }
});

modal.addEventListener('click', (e) => {
  if (e.target === modal) closeSearch();
});
```
