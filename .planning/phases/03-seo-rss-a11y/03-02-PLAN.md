Create `src/pages/rss.xml.ts` with the following exact implementation (from RESEARCH.md verified pattern):

```typescript
import rss from '@astrojs/rss';
import { getCollection } from 'astro:content';
import sanitizeHtml from 'sanitize-html';
import MarkdownIt from 'markdown-it';
import { SITE_TITLE, SITE_DESCRIPTION } from '../lib/consts';

const parser = new MarkdownIt();

export async function GET(context: { site: URL }) {
  const posts = await getCollection('posts', ({ data }) => !data.draft);
  return rss({
    title: SITE_TITLE,
    description: SITE_DESCRIPTION,
    site: context.site,
    customData: `<language>pt-BR</language>`,
    items: posts
      .sort((a, b) => b.data.pubDate.valueOf() - a.data.pubDate.valueOf())
      .map((post) => ({
        title: post.data.title,
        pubDate: post.data.pubDate,
        description: post.data.description,
        link: `/posts/${post.id.replace(/\.[^.]+$/, '')}/`,
        content: sanitizeHtml(parser.render(post.body ?? ''), {
          allowedTags: sanitizeHtml.defaults.allowedTags.concat(['img']),
        }),
      })),
  });
}
```

Key implementation notes:
- `!data.draft` filter is always active (unlike index.astro which has a PROD check) — RSS feed never includes draft posts per D-08
- `post.body` is the raw Markdown string from the content collection entry — NOT `compiledContent()` which does not exist on collection entries in Astro 6
- `markdown-it` renders raw Markdown to HTML, `sanitize-html` removes XSS vectors while keeping `<img>` tags
- `customData: '<language>pt-BR</language>'` injects the language tag per D-07
- `post.id.replace(/\.[^.]+$/, '')` strips the `.md` extension, same as index.astro
- `context.site` receives the `site:` value from astro.config.mjs at build time
- Trailing slash on `link:` is intentional (canonical URL consistency)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| post.body (Markdown) → RSS content | Raw Markdown rendered to HTML and embedded in RSS feed — untrusted if post content is compromised |
| robots.txt → search crawlers | Disallow rules affect indexing — over-restriction would hide content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | Tampering | RSS content via `post.body` | mitigate | `sanitize-html` with default allowlist + explicit `['img']` addition strips script tags, event handlers, dangerous attributes — XSS in RSS readers prevented |
| T-03-02-02 | Elevation of Privilege | robots.txt over-restriction | mitigate | `Allow: /` with no Disallow lines — entire site publicly indexable; minimal configuration prevents accidental blocking |
| T-03-02-03 | Spoofing | sitemap-index.xml URL in robots.txt | accept | URL is hardcoded to `https://sertaoseracloud.com/sitemap-index.xml` — matches the only valid deployment target; no dynamic computation needed |
| T-03-02-04 | Information Disclosure | RSS full content feed | accept | Full-content RSS is a deliberate product decision per D-06; target audience (senior devs) prefers full content in readers; no PII is exposed |
</threat_model>
